# アイテムシステム仕様書

## 概要
概念Pガチャから排出されるアイテム。使用時にNPC/村に効果を与える。
アイテムはインベントリに保持し、任意のタイミングで使用可能。

## ガチャ仕様
- **通貨**: 概念P
- **コスト**: 単発10P / 10連90P（キャラガチャと同額）
- **排出率**: N 70% / R 30%

---

## Nアイテム（7種）

### 💬 強制会話
- **効果**: 指定した2人のNPCを強制的に会話させる
- **使用時**: NPC2人を選択 → 次のAIループで会話ラリーを強制発火
- **実装ポイント**:
  - useAILoopに「強制会話キュー」を追加
  - キューにペアが入っていたら、通常の遭遇抽選をスキップして会話ラリーを実行
  - 場所は2人の中間地点（移動させてから会話？それともその場で？）
  - つまりポイント: 2人が離れている場合の移動演出をどうするか

### 🔍 発見率UP
- **効果**: 3日間（ゲーム内）civilization_eventの発生確率UP
- **実装ポイント**:
  - グローバルなバフstateが必要（有効期限: gameTime.day + 3）
  - AI応答パース時にcivilization_eventが含まれる確率は制御不能（AI任せ）
  - → プロンプトに「今日は特別な日だ。新しい発見がありそうな予感がする」を追加する方式
  - つまりポイント: バフのon/off管理とプロンプト注入の仕組み

### 🎨 髪色変更
- **効果**: 指定NPCの髪色（npc.color）をランダム変更
- **使用時**: NPC1人を選択 → COLORSからランダムに新色を選択 → npc.color更新
- **実装ポイント**:
  - 現在の色と同じ色が選ばれないようにする
  - 特にブロッカーなし、シンプル

### 📝 名前変更
- **効果**: 指定NPCの名前を変更
- **使用時**: NPC1人を選択 → テキスト入力 → npc.name更新
- **実装ポイント**:
  - 既存のNPC名と重複チェック
  - 歴史やログに旧名が残るが、それは仕様（過去の記録は変わらない）
  - 関係性のラベルなど、他NPCのmemory/longTermMemoryに旧名が残る問題
  - つまりポイント: 旧名の参照をどこまで書き換えるか（全部は非現実的→放置が妥当）

### 🕊️ 和解促進
- **効果**: 指定2人の好感度を+3
- **使用時**: NPC2人を選択 → 双方のrelationships[相手ID].scoreに+3
- **実装ポイント**:
  - 関係がない（relationships未登録）場合は新規作成（label: '知り合い', score: 3）
  - シンプル、ブロッカーなし

### ⚧️ 性別変更
- **効果**: 指定NPCの性別を反転（male↔female）
- **使用時**: NPC1人を選択 → gender反転 → スプライト再描画
- **制限**: SR/RでgenderLockが設定されているキャラは使用不可
- **実装ポイント**:
  - genderLockの判定にはgachaDataのテンプレートを参照する必要がある
  - しかしNPC自体にはgenderLock情報がない → roleからR_TEMPLATES/SR_TEMPLATESを逆引き
  - つまりポイント: genderLock判定のロジック。NPCにgenderLockフラグを持たせるか、テンプレート逆引きするか
  - 性別変更で見た目パーツ（body/cloth/eye/mouth/hair）が全部変わる
  - IDベースでパーツ決定してるので、性別変わっても同じIDで別パーツが選ばれる → 自然に切り替わる

### 💕 告白促進
- **効果**: 指定2人の次回会話時にプロンプトに告白衝動を注入
- **使用時**: NPC2人を選択 → 次回その2人が会話する際のプロンプトに追加
- **プロンプト追加文**: 「あなたは{相手名}に対して言葉にできない特別な感情を抱いている。今日こそ伝えたい。」
- **結果はAI次第**（好感度が低ければ拒否される可能性あり）
- **実装ポイント**:
  - 「次回の特定ペアの会話に1回だけ注入する」仕組みが必要
  - ささやきシステムの「whisperTask」に似た仕組み
  - つまりポイント: NPCに「confessionTarget」フラグを持たせる？それともグローバルなキューで管理？

## Rアイテム（4種）

### ☀️ ポジティブバフ
- **効果**: 7日間（ゲーム内）全NPCのプロンプトに前向き補正を追加
- **プロンプト追加文**: 「今日はなぜか穏やかな気分だ。相手の良いところを見つけたい。争いは避けたい。」
- **実装ポイント**:
  - グローバルバフstate（有効期限管理）
  - promptBuilder.tsで全NPCのプロンプト生成時にバフをチェックして注入
  - 発見率UPと同じバフ管理基盤を共有可能
  - つまりポイント: バフ管理の汎用的な仕組みを先に作るべき

### 💛 好感度ボーナス
- **効果**: 7日間、会話ラリーのrel_change.score_deltaに+2の補正
- **実装ポイント**:
  - useAILoop.tsのrel_change処理部分で、バフが有効ならdeltaに加算
  - マイナスのdeltaにも+2されるので、-5が-3になる（完全に打ち消しはしない）
  - つまりポイント: バフ管理基盤（ポジティブバフと共通）

### 🌪️ 天変地異
- **効果**: ランダムイベントを即発動
- **イベント例**: 嵐、豊作、金脈発見、謎の旅人、火事、疫病
- **実装ポイント**:
  - 既存の「神の介入」イベントシステムがあれば流用
  - なければ、全NPCのmemoryに「{イベント名}が起きた」を追加 + ログ表示
  - つまりポイント: イベントの種類と効果の定義。既存の天変地異システムの有無確認

### 🌾 豊作祈願
- **効果**: 未収穫の図鑑アイテム数 × 1Pを物理Pとして即時獲得
- **実装ポイント**:
  - civilizations配列からharvestedIdsに含まれないアイテムをカウント
  - カウント × 1を物理Pに加算
  - 0件なら「収穫するものがありません」で使用不可
  - つまりポイント: 特になし、シンプル

## 共通実装事項

### アイテムインベントリ
- `localStorage`に`kamisama_items`として保存
- 型: `Record<string, number>`（アイテムID → 所持数）
- UIは設定画面 or 専用タブで表示

### バフ管理基盤
- 発見率UP、ポジティブバフ、好感度ボーナスは共通の仕組みで管理
- 型: `Array<{ type: string; expiresDay: number }>`
- `localStorage`に`kamisama_buffs`として保存
- gameTime.dayと比較して期限切れを自動削除
- promptBuilder.tsとuseAILoop.tsからバフ一覧を参照

### 概念ガチャUI
- キャラガチャとは別のモーダル（または同じモーダル内のタブ切替）
- 演出はキャラガチャより簡素でOK（アイテムアイコン + レアリティ表示）
- 10連時にR確定枠は設けない（キャラガチャのSR確定は期間限定なので別扱い）

### アイテム使用UI
- インベントリ画面でアイテムをタップ → 対象選択（1人 or 2人 or 即発動）→ 確認 → 効果適用
- 使用後にログとバナーで通知

### つまりやすいポイントまとめ
1. **バフ管理基盤**: 複数のバフを統一管理する仕組みを最初に作る
2. **プロンプト注入**: promptBuilder.tsにバフやアイテム効果をどう組み込むか
3. **告白促進のキュー管理**: ささやきシステムとの設計の類似性と違い
4. **概念ガチャの演出**: キャラガチャの演出をどこまで流用するか
5. **性別変更のgenderLock判定**: NPC自体にフラグを持たせるか逆引きするか
6. **強制会話の移動**: 2人を近くに移動させるか、距離無視で会話させるか
