# あなたが神になる（You Become God）仕様書 v1.0

**アプリ名**: 高天原より常世のくにへ
**英語名**: From Takama-ga-hara to the Eternal Land
**プロジェクトフォルダ名**: kamisama
**略称**: 常世のくに / たかとこ

## 概要

AIが駆動するNPCたちが自律的に行動し、会話し、関係を築いていく様子を眺める「箱庭文明シミュレーター」。
プレイヤーは神様視点で村を観察し、時には介入してNPCたちの社会がどう変化するかを実験できる。

**コンセプト**: 「シムピープル × AIエージェント」— NPCの思考をLLMが担い、毎回異なるドラマが生まれる

### Groq API 仕様

- エンドポイント: `https://api.groq.com/openai/v1/chat/completions`
- 認証: `Authorization: Bearer <API_KEY>` ヘッダー
- リクエスト形式: OpenAI互換（`model`, `messages`, `max_tokens`, `temperature`）
- 無料枠制限: 30 RPM / 14,400 RPD（`llama-3.1-8b-instant`の場合）
- **APIキーはユーザーがブラウザ上で入力し、localStorageに保存する（サーバーには送信しない）**

```javascript
// リクエスト例
const response = await fetch("https://api.groq.com/openai/v1/chat/completions", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "Authorization": `Bearer ${apiKey}`
  },
  body: JSON.stringify({
    model: "llama-3.1-8b-instant",
    messages: [{ role: "user", content: prompt }],
    max_tokens: 200,
    temperature: 0.8
  })
});
```

## 時間システム

### 基本設計

- **現実1分 = ゲーム内1時間**
- 現実24分でゲーム内1日（24時間）が経過
- 現実1時間で約2.5日分の村の生活が展開される
- ゲーム内の日付・時刻はヘッダーに常時表示: `Day 3 / 14:32`

### 時間帯の定義

| 時間帯 | ゲーム内時刻 | 背景色調 | NPC行動傾向 |
|--------|------------|---------|------------|
| 早朝 | 5:00〜7:00 | 薄紫→オレンジ | 起床、準備 |
| 朝 | 7:00〜9:00 | 明るい暖色 | 朝の挨拶、移動開始 |
| 昼 | 9:00〜17:00 | 明るい緑・青 | 仕事、取引、会話が最も活発 |
| 夕方 | 17:00〜20:00 | オレンジ→赤紫 | 帰宅、夕食、1日の振り返り |
| 夜 | 20:00〜23:00 | 暗い紫・紺 | 活動縮小、内省的な独り言 |
| 深夜 | 23:00〜5:00 | 最も暗い | ほぼ活動停止。稀に夜更かし組 |

### 背景の変化

- 時間帯に応じてワールドマップの背景色をグラデーションで変化させる
- CSSグラデーションまたはオーバーレイの色・透明度を時刻に連動
- 変化はなめらかに（急にパッと変わらない）

### 時間帯がNPC行動に与える影響

プロンプトに現在のゲーム内時刻と時間帯を含める:

```
現在時刻: Day3 午前3時（深夜）
```

これにより、AIが時間帯に応じた自然な行動を返す（深夜に「市場で取引だ！」とは言わなくなる）。

加えて、時間帯による行動制御をフロントエンド側でも行う:

- **深夜（23:00〜5:00）**: 
  - NPC移動速度を50%に低下（眠そうにゆっくり動く）
  - 遭遇抽選の基本確率を10%に低下（ほとんど会話しない）
  - AI呼び出し間隔を2倍に延長（独り言も減る）
  - 多くのNPCは自宅または施設内で停止（移動目的地が民家固定になる確率80%）
- **昼（9:00〜17:00）**:
  - 通常の行動パラメータ
  - 遭遇抽選の基本確率+10%ボーナス（活発な時間帯）
- **夕方（17:00〜20:00）**:
  - 独り言の内容が「1日の振り返り」寄りになるようプロンプトに誘導
  - `現在時刻: 夕方。今日1日を振り返る時間帯。`

### 日付の経過

- 日付が変わる（ゲーム内0:00）タイミングで、ログに日付区切りを表示:
  ```
  ──── Day 4 ────
  ```
- 日付はlocalStorageにも保存し、オフラインログ生成にも使用

### 速度倍率との連携

- x1: 現実1分 = ゲーム内1時間（標準）
- x2: 現実1分 = ゲーム内2時間（1日が12分で経過）
- x3: 現実1分 = ゲーム内3時間（1日が8分で経過）

## NPC（住民）システム

### NPCデータ構造

```typescript
interface NPC {
  id: string;                    // 一意のID
  name: string;                  // 名前
  role: string;                  // 職業
  gender: "male" | "female";     // 性別
  personality: string;           // 性格（AIプロンプトに使用）
  emoji: string;                 // 表示用絵文字
  color: string;                 // テーマカラー（hex）

  // 位置
  x: number;
  y: number;
  targetX: number;               // 移動先X
  targetY: number;               // 移動先Y

  // パラメータ（トータル100ポイント制）
  params: {
    logic: number;               // 論理的思考（0-100）
    creativity: number;          // 創造性（0-100）
    morality: number;            // 道徳観（0-100）
    empathy: number;             // 共感力（0-100）
    ambition: number;            // 野心（0-100）
    sociability: number;         // 社交性（0-100）
  };

  // 状態
  mood: string;                  // 現在の気分
  memory: string[];              // 記憶（最新N件を保持）
  relationships: Record<string, {
    label: string;               // 関係ラベル（例: 「信頼」「警戒」「恩人」）
    score: number;               // 好感度（-100 〜 +100）
  }>;

  // 創発システム
  beliefs: string[];             // 信念・信仰（AIが記憶から自発的に追加）
  proposals: string[];           // 提案したルールや組織（AIが生成）

  // 内部制御
  bubble: string | null;         // 現在の吹き出しテキスト
  bubbleType: "say" | "think" | null;
  bubbleTimer: number;           // 吹き出し残り表示時間
  lastAiCall: number;            // 最後のAI呼び出しタイムスタンプ
  isWaiting: boolean;            // AI応答待ち中フラグ
}
```

### 自律行動のためのレイヤー設計

NPCの行動は以下の5レイヤーで構成され、上から順にプロンプトに埋め込まれる。
スクリプトで「何をしろ」とは指示せず、「何を求めているか」だけを与えることで、行動はAIに委ねる。
文明や制度はシステムが定義するのではなく、AIの会話から自然に生まれる。

#### レイヤー0: 世界の前提（全NPC共通・固定）

この世界がゼロから始まることを明示し、現代知識の持ち込みを防ぐ。

```
【この世界の前提】
あなたたちは何もない未開の地に降り立ったばかりの住民である。
この世界にはまだ以下のものが存在しない:
- 通貨やお金の概念
- 法律やルール
- 宗教や信仰体系
- 政治や統治の仕組み
- 文字や記録の方法
- 所有権の概念

あなたは現代の知識を持たない。
「お金」「法律」「政府」「宗教」といった言葉を知らない。
ただし、生活の中で必要に迫られたとき、
自分たちで新しい仕組みを考え出すことができる。

例えば:
- 食料が足りなくなれば「分け合う方法」を考えるかもしれない
- 争いが起きれば「決まりごと」を作るかもしれない
- 不思議な体験をすれば「目に見えない力」を信じるかもしれない
- 物を交換するとき「何かで数を揃える」ことを思いつくかもしれない

これらは強制ではない。必要だと感じたときだけ、自然に生まれるものである。

{ここに「村の発明・発見リスト」が動的に挿入される（後述）}
上記の発明・発見は、この村で既に生まれた概念である。
これらは使ってよいが、まだ生まれていない概念は使ってはならない。
```

#### レイヤー1: 本能（全NPC共通・固定）

全NPCのプロンプトに必ず含める基本欲求:

```
【本能】あなたは以下の欲求を常に持っている:
- 安全でいたい（危険を避けたい）
- 仲間がほしい（孤独は不安）
- 退屈は嫌だ（刺激や変化を求める）
- 自分の価値を認められたい（尊敬・感謝されたい）
- 世界を理解したい（なぜ自分はここにいるのか）
```

#### レイヤー2: 性格（パラメータから自動生成）

6つのパラメータ値に応じた性格修飾文。（詳細は「パラメータのAIプロンプトへの反映」セクションを参照）

#### レイヤー3: 経験（記憶の蓄積）

`memory` 配列の内容をプロンプトに含める。記憶が増えるにつれ、NPC は過去の経験に基づいて判断するようになる。
例: 「アヤメに騙された」記憶 → 次にアヤメに会ったとき警戒する

#### レイヤー4: 創発（AIに許可を与える）

プロンプトに以下を含めることで、AIが自発的に社会構造を生み出せるようにする:

```
【創発】あなたは経験や思考から、以下のことを自発的に行える:
- 新しい仕組みを考え出す（物の交換方法、役割分担など）
- 信じるものを見つける（自然への畏敬、目に見えない力など）
- 決まりごとを提案する（争いを防ぐ約束、分配の方法など）
- 他の住民を説得・勧誘する
- 仲間を集める
- 物を交換する条件を考える
これらは強制ではない。あなたの性格と経験から自然にそう思ったときだけ行動せよ。
```

AIの応答にこれらの創発的行動が含まれていた場合、`beliefs` や `proposals` に記録し、以降のプロンプトにも含める。
さらに、文明的な新概念が含まれていた場合は「村の発明・発見リスト」にも記録する。

## 文明トラッキングシステム

### 概要

NPCの会話から自然に生まれた概念・仕組み・信仰などを記録し、
「この村にどんな文明が生まれたか」を追跡する。
システムが文明を定義するのではなく、AIの発言から抽出して記録し、
それを次のプロンプトにフィードバックするループによって、文明はAIの中にだけ存在する。

### 村の発明・発見リスト

```typescript
interface CivilizationItem {
  id: string;
  day: number;                    // 発見されたゲーム内日付
  category: "exchange" | "rule" | "belief" | "organization" | "technology" | "custom";
  name: string;                   // 概念の名前（AI生成）
  description: string;            // 概要（30字以内）
  proposedBy: string;             // 提案者のNPC名
  status: "proposed" | "adopted" | "abandoned";  // 提案中 / 定着 / 廃れた
}
```

カテゴリの例:

| カテゴリ | 例 |
|---------|-----|
| exchange | 「野菜と道具の交換」「貝殻で数を揃える」 |
| rule | 「鉱山は順番に使う」「夜は静かにする」 |
| belief | 「森の大樹に力が宿る」「雨は天の恵み」 |
| organization | 「畑の世話係」「見張り番」 |
| technology | 「石で道具を作る」「火を囲んで集まる」 |
| custom | 「朝に広場で顔を合わせる」「収穫を分け合う」 |

### 発明・発見の検出

AI応答を解析して、新しい文明的概念が含まれているか判定する。

**方法1: AIの応答フォーマットに含める**

会話ラリーや独白のJSON応答に、オプションフィールドを追加:

```json
{
  "conversation": [...],
  "civilization_event": {
    "type": "exchange",
    "name": "野菜と祈りの交換",
    "description": "タケシが野菜を渡し、ゲンジが祈りを捧げる"
  }
}
```

`civilization_event` が含まれていた場合のみ、発明・発見リストに記録。

**方法2: 定期判定（バッチ）**

ゲーム内1日ごとに、その日の会話ログをまとめてAIに送り、
「この中に新しい文明的概念の萌芽があるか？」と判定させる。

```
以下はこの村の1日の出来事である。
この中に、新しい仕組み・信仰・慣習・技術の萌芽があれば抽出せよ。
なければ空配列を返せ。
{"discoveries":[{"type":"カテゴリ","name":"名前","description":"説明"}]}
```

### 文明の進化と退化

- **定着**: 同じ概念が複数のNPCの会話に繰り返し登場 → statusを `adopted` に変更
- **退化**: 一定期間（ゲーム内10日）言及されなくなった → statusを `abandoned` に変更
- **進化**: 既存概念の発展形が現れた場合、新しいアイテムとして記録し、元の概念への参照を残す
  - 例: 「野菜と道具の交換」→ 「貝殻で数を揃える」→ 「貝殻を全員が認める価値の基準にする」

### プロンプトへの反映

レイヤー0（世界の前提）内に、現在 `adopted` 状態の発明・発見を動的に挿入する:

```
【この村で生まれた概念】
- 物々交換: 野菜と道具を直接やりとりする（Day3〜）
- 森の畏敬: 森の大樹には不思議な力があると一部が信じている（Day12〜）
```

`abandoned` の概念は含めない（忘れ去られた設定）。ただし村の歴史には記録が残る。

### UIでの表示

- メイン画面に「文明パネル」タブを追加
- 発明・発見リストをカテゴリ別にアイコン付きで表示
- 各アイテムのステータス（提案中/定着/廃れた）を色分け
- タイムラインビュー: 文明の発展を時系列で可視化

## 住民追加機能

### 自動追加モード

- 「住民を追加」ボタン → 「自動生成」を選択
- 名前、職業、性格、絵文字、パラメータがすべてランダム生成
- パラメータはトータル100ポイントをランダム配分
- 即座にワールドに追加される

### 詳細設定モード

- 「住民を追加」ボタン → 「詳細設定」を選択
- 入力項目:
  - 名前（テキスト入力）
  - 性別（男 / 女）
  - 職業（テキスト入力 or プリセット選択）
  - 性格（テキストエリア、自由記述）
  - 絵文字（絵文字ピッカー or テキスト入力）
  - テーマカラー（カラーピッカー）

### パラメータ割り振りシステム

- **トータルポイント: 100**（固定）
- 6つのパラメータにスライダーで割り振る
- 残りポイントをリアルタイム表示
- 各パラメータ: 0〜100の範囲（ただしトータル100を超えられない）
- パラメータの説明ツールチップ:

| パラメータ | 高いと | 低いと |
|-----------|--------|--------|
| 論理的思考 | 合理的に判断する | 感情や直感で動く |
| 創造性 | 予想外の行動をとる | 定型的・保守的に動く |
| 道徳観 | 正義感が強い、嘘をつかない | 自分本位、手段を選ばない |
| 共感力 | 他者を助ける、気を遣う | 冷淡、他者に無関心 |
| 野心 | 権力や影響力を求める | 現状維持で満足する |
| 社交性 | 積極的に人と関わる | 孤立しがち、一人を好む |

### パラメータのAIプロンプトへの反映

パラメータ値に応じてプロンプトに性格修飾を追加する。

```
例: morality=90 の場合
→ プロンプトに追加: 「非常に強い倫理観を持ち、不正を絶対に許さない。嘘をつくことができない。」

例: morality=10 の場合
→ プロンプトに追加: 「自分の利益のためなら嘘も裏切りも厭わない。罪悪感をほとんど感じない。」

例: ambition=95, sociability=20 の場合
→ プロンプトに追加: 「極めて野心的で権力を渇望するが、人付き合いが苦手で孤独に策謀を巡らせる。」
```

各パラメータの閾値ごとのプロンプト文言例:

- **0-15**: 極端に低い → 「まったく〜ない」「極端に〜を欠く」
- **16-35**: 低い → 「あまり〜ない」「〜が苦手」
- **36-65**: 普通 → 特に言及しない（デフォルト）
- **66-85**: 高い → 「〜を重視する」「〜が得意」
- **86-100**: 極端に高い → 「非常に強い〜」「〜に突き動かされる」

## AI思考システム

### 基本フロー（シームレス・非同期）

1. NPCたちは常にワールド上を自律的に移動する（AI関係なし、フロントエンドのアニメーション）
2. NPC同士が接近（ENCOUNTER_DIST以内）した場合 →「遭遇抽選」を実行
3. 抽選に当選した場合のみ、会話ラリーのAPIリクエストを送信
4. 抽選にハズレた場合は、定型セリフ（API不使用）を吹き出し表示
5. 1人でいるNPCには一定間隔で独白プロンプトを送信
6. AIの応答を待っている間も、画面上のNPCは動き続ける（待ちが発生しない）

### 遭遇抽選システム

NPCが接近しても必ず会話が発生するわけではない。抽選で会話イベントの発生を判定する。

#### 抽選確率の計算

```
基本確率: 30%

加算補正:
+ (A社交性 + B社交性) / 2 * 0.3   — 社交的な者同士ほど話しやすい
+ |関係性スコア| * 0.2              — 仲良しでも敵対でも絡みやすい（無関心が一番スルー）
+ イベント発生中 ? 15 : 0           — 天変地異中は会話が生まれやすい

減算補正:
- 直近60秒以内に同じペアで会話済み ? 50 : 0  — さっき話したばかりならスルー

最終確率: 0〜90%にクランプ（最低限のスルーと最大限の発生を保証）
```

#### 抽選ハズレ時 — 定型セリフ（API不使用）

会話イベントが発生しなかった場合、配列からランダムに選んだ定型セリフを吹き出し表示する（1.5秒間）。
セリフは関係性・パラメータに応じて変化する。

**通常（関係性なし or 中立）**:
```
["おっと", "やあ", "よう", "（会釈）", "..."]
```

**友好（好感度 +30以上）**:
```
["よっ！", "元気？", "また会ったね！", "おーい！", "いい天気だね"]
```

**敵対（好感度 -30以下）**:
```
["...チッ", "（目を逸らす）", "邪魔だよ", "（無視）", "...ふん"]
```

**社交性が極端に低い（15以下）場合**:
上記に関わらず以下から選択:
```
["...", "（無言で通り過ぎる）", "（うつむく）"]
```

**ログ表示**: 定型セリフ時もライブログに記録するが、薄いグレーで表示して会話イベントと区別する。
**クールダウン**: 同じペアのすれ違い演出は最低10秒間隔を空ける（吹き出しが出すぎるのを防ぐ）。

### 移動ロジック（AI非依存）

- 各NPCは `targetX`, `targetY` を持ち、毎フレームそこに向かって移動する
- 目的地に到着したら、ランダムに次の目的地を選ぶ:
  - 60%の確率: 施設のいずれかの近辺
  - 40%の確率: マップ上のランダムな位置
- 移動速度: 1.2px/tick（速度倍率で変動）

### プロンプト設計

#### 遭遇プロンプト（近くに他NPCがいる場合）— 会話ラリー方式

1回のAPI呼び出しで2〜4往復の会話を生成する。これにより自然な対話のやりとりが生まれる。

```
村シムの会話生成。JSONのみ返せ。
A:{名前A}({職業A}){性格文A}{パラメータ由来の性格修飾A}
B:{名前B}({職業B}){性格文B}{パラメータ由来の性格修飾B}
場所:{現在地の施設名}
{AのBに対する関係性があれば記載}
{BのAに対する関係性があれば記載}
{Aの記憶があれば: A記憶:{最新3件}}
{Bの記憶があれば: B記憶:{最新3件}}
2人が出会った。2〜4往復の自然な短い会話を生成せよ。各セリフは25字以内。
{"conversation":[{"who":"名前","say":"セリフ"},...],"summary":"会話の要約15字","a_mood":"Aの気分4字","b_mood":"Bの気分4字","rel_change":{"a_to_b":{"label":"関係8字","score_delta":数値},"b_to_a":{"label":"関係8字","score_delta":数値}}}
```

**会話ラリーの表示**: 会話の各セリフを1〜2秒間隔で順番に吹き出し表示する。AとBの吹き出しが交互にポンポン出てくる演出。

#### 独白プロンプト（1人でいる場合）

```
村シムNPC。JSONのみ返せ。
あなた:{名前}({職業}){性格文}{パラメータ由来の性格修飾}
場所:{現在地の施設名}
{記憶があれば: 記憶:{最新3件}}
1人でいる。何を思い何をする？
{"think":"独り言20字以内","action":"行動12字以内","mood":"気分4字"}
```

### AI応答の処理

1. JSONパース（失敗時はそのターンをスキップ）
2. 吹き出し表示:
   - 遭遇時: `conversation` 配列のセリフを1〜2秒間隔で順番に💬吹き出し表示
   - 独白時: `think` を💭吹き出しで表示（3秒間）
3. ログに追加（時刻、名前、行動、発言、内心）
4. NPC状態更新:
   - `mood` を更新（遭遇時はA・B両方）
   - `memory` に今回の出来事を追加（最新6件を保持）
   - 遭遇時: 双方の `relationships` を `rel_change` に基づいて更新
5. 相手NPCの記憶にも出来事を追加

### API呼び出しの制御 — 並列パイプライン

**基本方針**: 常に2〜3本のAPIリクエストを並列で走らせ、画面上に同時多発的にドラマが生まれるようにする。

- **並列数**: 最大2〜3リクエスト同時（Groq無料枠30RPMに対して余裕を持たせる）
- **優先度**: 遭遇（会話ラリー）> 独白
- **パイプライン動作**:
  1. AとBが遭遇 → 会話ラリーのAPIリクエストを投げる
  2. 同時に、別の場所にいるCの独白リクエストも投げる
  3. どちらかが返ってきたら即座に画面に反映し、次の対象を探して新しいリクエストを投げる
  4. 常にリクエストが飛んでいる状態を維持
- **レート制限**: 1分あたりのリクエスト数を内部カウントし、25回を超えたらペースを落とす（30RPMの上限に対してマージン確保）
- 各NPCの最終呼び出しから `AI_INTERVAL_MS`（デフォルト10秒）経過後に次の呼び出し対象となる
- 速度倍率（x1/x2/x3）でインターバルも短縮: `AI_INTERVAL_MS / speed`
- レート制限エラー（429）を受けた場合、30秒間API呼び出しを休止し、ユーザーに通知

## 記憶・歴史システム

### 概要

NPCの記憶と村の歴史を階層的に管理し、文明の進化を長期的に追跡できるようにする。
個々の出来事は時間とともに圧縮されるが、大きな流れは永続的に保持される。

### 3層構造

```
┌─────────────────────────────────────────┐
│  村の歴史（全NPC共有）                     │
│  ├── 過去の時代（圧縮済み）3〜5件         │
│  │   "【黎明期 Day1-15】村の基盤が..."    │
│  └── 最近の歴史（詳細）直近10〜15件        │
│       "Day32: アヤメが通貨制度を提案"     │
├─────────────────────────────────────────┤
│  個人の長期記憶（NPC個別）最大20件         │
│  "アヤメは信用できない（Day5の買収事件）"  │
│  "リンは不器用だが信頼できる"             │
├─────────────────────────────────────────┤
│  個人の短期記憶（NPC個別）直近6件          │
│  "さっきリンと畑で話した"                │
└─────────────────────────────────────────┘
```

### レイヤー1: 短期記憶（個人）

- 保持件数: 最新6件（FIFOで古いものから削除）
- 記憶フォーマット: 自然言語の短い文
  - 遭遇時: `「{相手名}に会い「{発言}」と言った({場所})」`
  - 独白時: `「{場所}で{行動}」`
  - イベント時: `「天変地異:{イベント内容}」`
- 毎回のAIプロンプトに全件含める
- UIでは住民詳細パネルに表示

### レイヤー2: 長期記憶（個人）

- 保持件数: 最大20件
- 短期記憶から「重要な出来事」を昇格させて保存
- 昇格の判定: AI会話ラリーの応答に `rel_change` が含まれる場合（関係性が変化した出来事）、または神の介入イベント発生時に自動昇格
- フォーマット: `"(Day{X}) {要約}"` — 日付つきで圧縮
  - 例: `"(Day5) アヤメに畑の買収を持ちかけられ断った"`
  - 例: `"(Day12) リンが剣を作ってくれた"`
- 20件を超えた場合、最も古いものから削除
- AIプロンプトには全件含める

### レイヤー3: 村の歴史（全NPC共有）

村全体の共有知識。全NPCのプロンプトに共通で含める。

#### 最近の歴史（詳細）

- 保持件数: 10〜15件
- フォーマット: `"Day{X}: {出来事の要約}"`
  - 例: `"Day3: タケシとゲンジの間で食料と祈りの交換が始まった"`
  - 例: `"Day8: アヤメが市場での取引ルールを提案した"`
  - 例: `"Day15: 嵐が村を襲い、畑が荒れた"`

#### 過去の時代（圧縮済み）

- 保持件数: 3〜5件（時代ごとに1エントリ）
- 最近の歴史が15件に達したら「時代の圧縮」を実行
- フォーマット: 2〜3行の要約テキスト
  - 例:
    ```
    【黎明期 Day1-15】
    村の基盤が築かれた時代。タケシの畑が食料源となり、
    ゲンジが信仰を広め始めたが賛同者は少なかった。
    アヤメは各住民に取引を持ちかけ、商業の基礎を作った。
    ```

### 時代の圧縮プロセス

最近の歴史が15件に到達したとき、自動的に圧縮を実行する。

```
村シムの歴史圧縮。JSONのみ返せ。

以下の出来事群を1つの「時代」として要約せよ。
時代名と、2〜3行の要約を生成すること。
重要な転換点、人間関係の変化、社会の発展を中心にまとめよ。

【出来事】
- Day1: タケシが畑を耕し始めた
- Day2: ゲンジが布教を始めた
- Day3: アヤメがリンに取引を持ちかけた
...

{"era_name":"時代名（例:黎明期）","period":"Day1-15","summary":"2〜3行の要約"}
```

圧縮後:
- 要約が「過去の時代」に追加される
- 圧縮元の15件は「最近の歴史」からクリアされる
- API消費: 1回の呼び出し

### 歴史への記録判定

どの出来事を村の歴史に記録するかは、以下の2つの方法で判定する。

**自動記録（即時）**:
- 神の介入イベント発生時
- NPCが `proposals`（ルール提案）や `beliefs`（新しい信念）を生成したとき
- 関係性スコアが ±50 を超えたとき（強い友好 or 敵対の成立）

**AI判定（定期）**:
- ゲーム内1日（現実24分）ごとに1回、「直近の出来事から村の歴史に記録すべき重要な出来事を抽出せよ」というプロンプトを投げる
- API消費: 1回/日

### プロンプトへの組み込み

全レイヤーを以下の順序でプロンプトに含める:

```
【村の歴史】
〈過去の時代〉
- 【黎明期 Day1-15】村の基盤が築かれた時代。...
- 【発展期 Day16-30】取引が活発化し...
〈最近の出来事〉
- Day32: アヤメが通貨制度を提案した
- Day35: ゲンジの教会に初めての信者が現れた
...

【あなたの記憶】
〈重要な記憶〉
- (Day5) アヤメに畑の買収を持ちかけられ断った
- (Day12) リンが農具を直してくれた
〈最近の記憶〉
- さっきリンと畑で話した
- 井戸の水を汲んだ
```

### トークン数の見積もり

| レイヤー | 件数 | 1件あたり | 合計トークン目安 |
|---------|------|----------|---------------|
| 過去の時代 | 3〜5件 | 50〜80トークン | 150〜400 |
| 最近の歴史 | 10〜15件 | 15〜20トークン | 150〜300 |
| 長期記憶 | 20件 | 15トークン | 300 |
| 短期記憶 | 6件 | 15トークン | 90 |
| **合計** | | | **約700〜1100トークン** |

Qwen3-32Bの128Kコンテキストウィンドウに対して1%未満。余裕で収まる。

### データ保存

- 全レイヤーのデータはlocalStorageに保存
- オフラインログ生成時にも村の歴史を参照・更新する
- 村の歴史はUIの「歴史タブ」から閲覧可能（時系列で全時代を表示）

## 関係性システム

### データ構造

各NPCは他のNPCとの関係性を `relationships` として保持する。

```typescript
relationships: Record<string, {
  label: string;   // 「信頼」「警戒」「恩人」「商売仲間」「ライバル」等
  score: number;   // -100（敵対）〜 +100（深い信頼）
}>
```

### 更新ロジック

- AI応答の `rel_change` に基づいて更新:
  - `label`: 関係ラベルを上書き
  - `score_delta`: 現在スコアに加算（-100〜+100にクランプ）
- 相手側のrelationshipsにも逆方向の記録を自動追加（ラベルは相手のAI応答で決まる）

### 関係性ネットワーク図

- 住民間の関係性をネットワークグラフとして可視化
- ノード: 各NPCの絵文字 + 名前
- エッジ: 関係がある住民間に線を引く
  - 線の色: 好感度に応じて変化（赤=敵対、緑=友好、グレー=中立）
  - 線の太さ: 関係の強さ（|score|の大きさ）に応じて変化
  - ラベル: 関係名を線の中点に表示
- 表示切替: メイン画面にトグルボタンで表示/非表示を切り替え
- ネットワーク図はcanvasまたはSVGで描画（d3-forceなどのライブラリ使用も可）

| ⚡ 神の介入 | ランダムイベント発生 |
| 🗣️ ささやき | 特定NPCに神の声を送る（後述） |
| 👤 住民追加 | 住民追加モーダルを開く |
| 🔗 関係図 | 関係性ネットワーク図の表示/非表示トグル |
| 📜 文明 | 文明パネル（発明・発見リスト）の表示/非表示 |
| ⚙️ 設定 | APIキー変更等の設定画面 |

### ささやき機能（神の声）

プレイヤーが特定のNPCに自然言語で短いメッセージを送れる機能。
NPCに直接命令するのではなく、「神のささやき」としてAIプロンプトに混ぜる。
NPCは性格に応じて従うことも無視することもある。

#### 仕様

- **使用制限**: 10分に1回（クールダウンタイマーをUIに表示）
- **文字数**: 50文字以内
- **対象**: NPC1人を選択してから入力

#### 操作フロー

1. ヘッダーの「🗣️ ささやき」ボタンを押す（またはNPCをクリックして「ささやく」を選択）
2. 対象NPCを選択
3. テキスト入力（50文字以内）: 例「森に行ってみよ」「アヤメを信じるな」「仲間を集めよ」
4. 送信 → 次回のそのNPCのAI呼び出し時にプロンプトに反映
5. クールダウン開始（10分間は次のささやき不可）

#### プロンプトへの反映

NPCの次の独白または会話ラリー時に、プロンプトに以下を追加:

```
【神の声】「森に行ってみよ」
※これは天から聞こえたかすかな声である。
従うかどうかはあなたの性格と判断に委ねられる。
強制ではない。無視してもよい。
ただし、この声に対して必ず何らかの反応を示すこと（従う、疑う、怖がる、利用する、反発する等）。
反応をthinkまたはsayに含めること。
```

#### AI応答フォーマットの拡張

ささやき受信時、通常の応答に `whisper_reaction` フィールドが追加される:

```json
{
  "think": "...声が聞こえた。森に行けと...",
  "action": "森の方角を見つめる",
  "mood": "困惑",
  "whisper_reaction": "obey | doubt | fear | exploit | reject"
}
```

- `obey`: 従う（「導きに従おう」）
- `doubt`: 疑う（「本当にそうすべきか...？」）
- `fear`: 怖がる（「何者かの声が...怖い」）
- `exploit`: 利用する（「この声を使えば有利になるかも」）
- `reject`: 反発する（「指図されるいわれはない」）

`whisper_reaction` はログに表示し、長期記憶にも記録する。

#### NPCの反応パターン（性格による）

AIに委ねるが、傾向としては:

- **道徳観が高い**: 「神の導きだ…従おう」と素直に受け入れやすい
- **論理的思考が高い**: 「なぜそうすべきか？」と理由を考えてから判断
- **野心が高い**: 「この声を利用できないか？」と打算的に解釈
- **創造性が高い**: 「面白そうだ、やってみよう」と好奇心で動く
- **社交性が低い**: 「……聞こえなかったことにする」と無視しがち

#### ログ表示

ささやき送信時にログに記録:

```
[18:30:00] 🔮 神のささやき → タケシ
  「森に行ってみよ」
```

NPCの反応もログに記録（通常のAI応答として）:

```
[18:30:15] AI 👨‍🌾 タケシ
  💭 ...なにか声がした。森に行けと...
  → 不思議に思いながら森の方を見る
```

#### 制限の意図

- 10分に1回の制限により「一言一言を大切に使う」戦略性が生まれる
- 50文字制限により簡潔で象徴的な神託になる（長文で細かく指示できない）
- NPCが従うとは限らないため「思い通りにならない面白さ」が保たれる
- 神（プレイヤー）は万能ではなく、あくまで「ささやく」ことしかできない

### 神の認識段階システム

同じNPCにささやきを繰り返すと、そのNPCが徐々に「神の存在」を認識し始める。
プレイヤーの介入がNPCの人生を変えていく仕組み。

#### NPCデータへの追加

```typescript
interface NPC {
  // ...既存フィールド...
  whisperCount: number;          // 累計ささやき受信回数
  divineAwareness: "none" | "vague" | "sensing" | "aware" | "prophet";
}
```

#### 認識段階とプロンプトへの影響

| 段階 | ささやき回数 | NPCの状態 | プロンプトに追加する文 |
|------|-------------|-----------|---------------------|
| none | 0回 | 通常 | （なし） |
| vague | 1〜2回 | なんとなく気になる | 「最近、風の音に紛れて声が聞こえた気がする。気のせいだろう。」 |
| sensing | 3〜4回 | 感じ始めている | 「また声が聞こえた。誰かが自分を見ている気がする。時々空に向かって語りかけたくなる。」 |
| aware | 5〜7回 | 神を認識 | 「天から声が聞こえる。見えない何者かが自分に語りかけている。これは確信だ。独り言で神に語りかけることがある。」 |
| prophet | 8回以上 | 預言者化 | 「あなたは神の声を聞く者である。他の住民はこの事実を知らない。神の意志を伝えるべきか、秘めるべきか、あなた自身が判断せよ。」 |

#### 各段階での行動変化

**vague（1〜2回）**:
- 独白に「...風の音だったか？」「空が気になる」程度の描写が稀に混じる
- 他NPCとの会話には影響なし

**sensing（3〜4回）**:
- 独白に「また聞こえた...」「見えない何かが...」が混じり始める
- 時折、空を見上げる行動をとる
- 他NPCに「最近変な声が聞こえないか？」と聞くことがある

**aware（5〜7回）**:
- 独白が神への語りかけになることがある:「神よ、見ているのか...」「なぜ私に語りかける...」
- 他NPCに「天から声が聞こえる」と打ち明ける可能性
  - 信じてもらえない → 孤立
  - 信じてもらえる → 畏敬の目で見られる
- 信仰系の行動が増える（祈る、祠に行く等）

**prophet（8回以上）**:
- 「神の代弁者」として振る舞い始める可能性
- 他NPCに神の声の内容を伝える（正確に伝えるとは限らない）
- 村の中で特別な存在になる:
  - 崇拝される or 気味悪がられる or 利用される（アヤメなら利用しそう）
- 文明トラッキングに「預言者の出現」として記録
- 村の歴史に大きな影響を与える

#### ログ表示

認識段階が変化したとき、ログに通知:

```
[19:45:00] 🔮 ゲンジの神への認識が深まった（sensing → aware）
```

#### 注意事項

- 認識段階は下がらない（一度神を認識したら忘れない）
- 複数のNPCが同時にprophet段階になることもありえる → 預言者同士の対立が生まれる可能性
- 一度もささやかれていないNPCは完全にnone状態で、神の存在を知らない
- prophet段階のNPCが他NPCに神の話をすることで、ささやかれていないNPCにも間接的に神の概念が広まる可能性がある（文明トラッキングの「belief」として記録）

### 神の介入（イベント）

ランダムで以下のいずれかのイベントが発生:

| イベント | 効果 |
|----------|------|
| 🌪️ 嵐 | 「激しい嵐が村を襲った！」 |
| 💎 金脈発見 | 「鉱山で大量の金が見つかった！」 |
| 🐉 謎の獣 | 「森で巨大な獣が目撃された！」 |
| 🌸 豊作 | 「畑が大豊作！食料が溢れている！」 |
| 🔥 火事 | 「市場で火事が発生した！」 |
| 👑 選挙 | 「村のリーダーを選ぶ選挙が始まった！」 |

- イベント発生時、全NPCの記憶に追加
- ライブログにイベント通知を表示
- イベント表示バナーを画面上部に10〜15秒間表示

## API使用量表示とフォールバック

### ヘッダーのAPI残量表示

ヘッダーに常時表示するAPI使用量ウィジェット:

```
☁️ AI残量: 847/1,000 (あと約5時間遊べます)
[████████░░] 85%
```

#### 表示項目

- 今日の使用回数 / 1日の上限（1,000 RPD）
- 残りで遊べる推定時間（直近の消費ペースから算出）
- プログレスバー（残量に応じて色が変化）
  - 80%以上残: 緑
  - 50〜80%残: 黄色
  - 20〜50%残: オレンジ
  - 20%未満: 赤

#### 推定時間の算出

```
直近10分間のAPI呼び出し回数から1時間あたりの消費を推定
残り回数 ÷ 1時間あたり消費 = 残り時間（時間単位）
```

例: 残り800回、1時間あたり約150回消費 → 「あと約5時間遊べます」

### API残量が少なくなったときの動作

#### 残量50%以下（500回以下）

- 特に制限なし。表示色が黄色に変わるだけ
- 「💡 ヒント: AI呼び出し間隔を広げると長く遊べます」を一度だけ通知

#### 残量20%以下（200回以下）

- AI呼び出し間隔を自動的に1.5倍に延長（節約モード）
- ヘッダーに「🔋 節約モード」を表示
- ユーザーが設定で解除可能

#### 残量5%以下（50回以下）

- AI呼び出し間隔をさらに2倍に延長
- 「⚠️ 本日のAI残量がわずかです」を通知

#### 残量0（上限到達 or 429エラー受信）

- AI呼び出しを完全停止
- 定型セリフのみで村を回し続ける（NPCは歩き回り、すれ違い演出は継続）
- ヘッダーに「☕ AIが休憩中（明日リセットされます）」を表示
- ログに以下を表示:
  ```
  ──── ☕ 本日のAI枠を使い切りました ────
  村は静かな夜を迎えています...
  明日になればAIが再び目覚めます。
  ```
- 翌日0:00 UTC（RPDリセットタイミング）に自動的にAI呼び出しを再開

### データ管理

- 今日の使用回数はlocalStorageに保存
- 日付が変わったらカウントをリセット
- Groqのレスポンスヘッダー `x-ratelimit-remaining-requests` も参照し、より正確な残量を表示

## オフラインログ生成（おかえりシステム）

### 概要

ブラウザを閉じている間も「世界は動いていた」ように見せる仕組み。
再訪時に、不在中に起きた出来事をAIにまとめて生成させ、ログとして表示し、NPC状態を更新する。

### データ保存

- ブラウザを閉じる際（`beforeunload`イベント）、以下をlocalStorageに保存:
  - 全NPCの現在状態（位置、記憶、関係性、beliefs、mood等）
  - 最終アクセス時刻（タイムスタンプ）
  - 現在のワールド状態（発生中のイベント等）

### 再訪時の処理フロー

1. ブラウザを開く
2. localStorageから前回の状態と最終アクセス時刻を読み込む
3. 経過時間を計算（現在時刻 - 最終アクセス時刻）
4. 経過時間が **5分未満** → そのまま再開（オフラインログ生成なし）
5. 経過時間が **5分〜24時間** → オフラインログ生成を実行
6. 経過時間が **24時間超** → 最大24時間分として生成（あまりに長期間だと整合性が崩れるため）

### 「おかえり画面」

ログ生成中は専用の画面を表示:

```
┌─────────────────────────────────────┐
│                                     │
│     高天原より常世のくにへ            │
│                                     │
│     🌅 おかえりなさい、神様。         │
│                                     │
│     あなたがいない間に                │
│     6時間が経ちました。              │
│                                     │
│     村では様々なことが               │
│     起きていたようです...             │
│                                     │
│     ⏳ 記録を生成中...               │
│                                     │
└─────────────────────────────────────┘
```

### オフラインログ生成プロンプト

経過時間に応じてイベント数を調整:
- 5分〜1時間: 2〜3件
- 1〜6時間: 5〜8件
- 6〜24時間: 8〜12件

```
村シムの不在中ログ生成。JSONのみ返せ。
経過時間: {X時間Y分}

【住民一覧と現在状態】
{各NPCの名前、職業、性格、現在の気分、記憶、関係性、信念を列挙}

【指示】
この時間内に起きた主要な出来事を{N}件生成せよ。
- 各住民の性格・パラメータ・関係性に基づいた自然な展開にすること
- 時系列順に並べること
- 関係性の変化、新しい信念の発生、対立や協力を含めること
- 経過時間が長いほど大きな変化が起きてよい

{"events":[
  {
    "time_offset": "経過X分後",
    "who": ["関与したNPCの名前"],
    "type": "conversation|action|discovery|conflict|alliance|belief",
    "summary": "出来事の要約（30字以内）",
    "detail": "詳細な描写（60字以内）",
    "dialogue": "発言があれば（なければ空文字）",
    "memory_add": {"NPC名": "追加する記憶"},
    "relationship_changes": {"NPC名→NPC名": {"label": "関係", "score_delta": 数値}},
    "belief_add": {"NPC名": "新しい信念（あれば）"},
    "mood_update": {"NPC名": "新しい気分"}
  },
  ...
]}
```

### ログ表示の演出

生成されたイベントを「おかえり画面」上で順番に表示する:

- 各イベントが1〜2秒間隔でフェードインで表示される
- 巻物を読むようにスクロールしていく演出
- 全イベント表示後に「村を見に行く →」ボタンでメイン画面へ遷移

### NPC状態の反映

オフラインログの内容に基づいて、メイン画面に遷移する前にNPC状態を一括更新:

- `memory`: ログの各イベントから記憶を追加
- `relationships`: `relationship_changes` を反映
- `beliefs`: `belief_add` を反映
- `mood`: 最後のイベントの `mood_update` を反映

### API消費

- オフラインログ生成は1〜2回のAPI呼び出しで完了（イベント数が多い場合は2回に分割）
- 通常のリアルタイム再生と比べて圧倒的に効率が良い

## 補足・注意事項

- **APIキーの扱い**: サーバーには一切送信しない。ブラウザのlocalStorageに保存し、クライアントサイドからGroq APIを直接呼び出す。
- **エラーハンドリング**: API呼び出し失敗時はそのNPCの行動をスキップし、次のNPCに進む。UIにエラー表示はしない（自然に見せる）。
- **JSON パースエラー**: LLMの応答がJSON形式でない場合、```json ``` のフェンスを除去してリトライ。それでも失敗したらスキップ。
- **レート制限**: 429エラー受信時、30秒間AI呼び出しを停止し、ヘッダーに「⏳ API休憩中」と表示。
- **パフォーマンス**: NPC数が増えた場合（10人以上）、AI呼び出しインターバルを自動延長して制限内に収める。
- **将来拡張**: WebLLM対応、Ollama対応、村のルール設定（通貨制度等）、複数の村、住民の追放/死亡、建築システムなど。ただし初期バージョンでは実装しない。

## 将来拡張: アバタービルダー（Phase 6以降）

初期バージョンでは絵文字表示だが、将来的にPNGパーツベースのアバターシステムに移行する。

### 方式: グレースケールPNG + CSSカラーオーバーレイ

パーツはグレースケールのPNGで作成し、プログラム側で`mix-blend-mode: multiply`を使って色を動的にかぶせる。

- **白い部分**: 指定色が鮮やかに乗る
- **グレーの部分**: 影として暗めに色が乗る
- **黒い部分**: 輪郭線としてそのまま残る
- **背景**: 透過PNG

### パーツ構成

| パーツ | 種類数目安 | 色変更 |
|--------|-----------|--------|
| 体（ベース） | 1種 | 肌色をオーバーレイ |
| 髪型 | 10〜15種 | 髪色をオーバーレイ |
| 目 | 5〜8種 | 色変更なし（そのまま使用） |
| 服 | 8〜10種 | 服色をオーバーレイ |
| アクセサリ | 5種程度 | 任意で色変更 |

### レンダリング方式

```html
<div style="position: relative; width: 64px; height: 64px;">
  <!-- 体ベース -->
  <img src="body.png" style="position: absolute;" />
  <div style="position: absolute; inset: 0; background: #FFDAB9; mix-blend-mode: multiply;" />
  <!-- 服 -->
  <img src="clothes_01.png" style="position: absolute;" />
  <div style="position: absolute; inset: 0; background: #4CAF50; mix-blend-mode: multiply;" />
  <!-- 髪 -->
  <img src="hair_03.png" style="position: absolute;" />
  <div style="position: absolute; inset: 0; background: #8B4513; mix-blend-mode: multiply;" />
  <!-- 目（色変更なし） -->
  <img src="eyes_02.png" style="position: absolute;" />
</div>
```

### 住民追加時のカスタマイズ

- 各パーツをプレビュー付きで選択
- カラーピッカーで髪色・服色・肌色を自由に設定
- 自動生成モードではパーツ・色ともにランダム選択

## 将来拡張: 性格・職業の動的変化（Phase 7以降）

### 概要

NPCの6パラメータ、性格文、職業が固定ではなく、経験によって変化する。
「探検家だったソラが鉱夫になる」「穏やかだったタケシが強気になる」といった変化が自然に起きる。

### パラメータの変動

- 記憶や関係性の変化に応じて、パラメータが ±1〜3 ずつ変動
- トータルポイント（100）は維持される（何かが上がれば何かが下がる）
- 変動の判定: ゲーム内1日ごとに、その日の出来事からAIが判定

```
この住民の今日の経験に基づき、パラメータの変動を判定せよ。
変動がなければ空を返せ。変動は±1〜3の小さな値にすること。
トータル100を維持すること。
{"param_changes":{"sociability":+2,"ambition":-2}}
```

### 職業の変化

NPCの行動パターンが元の職業と乖離し始めたとき、AIが職業転向を提案する。

- **判定タイミング**: ゲーム内7日ごと
- **判定基準**: 直近の行動ログ・記憶を見て、現在の職業と実際の行動が合っているか

```
この住民の直近7日間の行動を見て、職業の変化が自然かどうか判定せよ。
変化なしなら空を返せ。
{"job_change":{"from":"探検家","to":"鉱夫","reason":"鉱石採掘に没頭するようになった"}}
```

- 職業が変わるとき、NPCの独白に変化の兆しが現れる:
  - 「最近、鉱石を掘ってる時間の方が長いな...」
  - 「もう冒険より石の方が面白い...」
- 職業が正式に変わったら、ログと村の歴史に記録:
  ```
  [Day25] 🔄 ソラの職業が変わった: 探検家 → 鉱夫
  「鉱石採掘に没頭するようになった」
  ```

### 性格文の更新

パラメータが大きく変化した場合（累計±15以上の変動）、性格文を再生成する。

```
以下のパラメータを持つNPCの性格文を生成せよ（50字以内）。
名前: リン / 職業: 職人 / 性別: 女
論理:18 創造:12 道徳:22 共感:25 野心:8 社交:15
過去の性格: 「無口で義理堅い鍛冶屋」
{"personality":"以前より穏やかになった鍛冶屋。仲間への気遣いが増えた。"}
```

### 例

- リンがタケシとの交流を重ねる → 社交性が+1ずつ上昇 → 性格文が「無口で義理堅い」から「以前より穏やかになった」に変化
- アヤメが裏切りを繰り返す → 周囲のNPCの道徳観が変化
- ゲンジの布教が失敗し続ける → 野心が低下、共感力が上昇 → 「権力欲がある聖職者」から「人々に寄り添う聖職者」に変化
- ソラが鉱石採掘にハマる → 職業が探検家から鉱夫に転向

### UI表示

- 住民詳細パネルに「パラメータ推移グラフ」を追加（折れ線グラフ）
- 変動があったときにログに「📊 リンの社交性が上がった」と表示

## 将来拡張: フィールド・建築システム（Phase 8以降）

### 概要

NPCがAIの判断で建物を建てたり、土地を開拓したりする。マップが時間とともに変化していく。

### 建築の仕組み

- NPCが独白や会話の中で「ここに小屋を建てたい」「新しい畑を開墾する」等の意思表示をした場合、建築イベントとして検出
- 建築にはゲーム内の日数が必要（小屋: 3日、大きな建物: 7日など）
- 建築中は該当NPCの行動が「建設作業」に固定される
- 完成するとマップ上に新しい施設が追加される

### 建築可能な施設

| 施設 | 建築日数 | 効果 |
|------|---------|------|
| 小屋 | 2日 | NPCの住居。帰る場所ができる |
| 新しい畑 | 3日 | 食料生産が増える |
| 商店 | 5日 | 取引の場が増える |
| 祠 | 4日 | 信仰の場。近くで信仰イベントが起きやすくなる |
| 見張り台 | 3日 | 周辺の安全度が上がる |
| 集会所 | 7日 | 複数人での会話イベントが発生する場所 |

### 土地の概念

- マップを格子状のエリアに分割
- 各エリアに「所有者」を設定可能
- NPCが「ここは自分の場所だ」と主張 → 所有の萌芽
- 他のNPCが同意するか反発するかで土地制度が生まれる
- 土地の売買も文明トラッキングの一部としてAIに委ねる

## 将来拡張: 寿命・世代交代（Phase 9以降）

### 概要

NPCに寿命を導入し、死と誕生のサイクルによる世代交代を実現する。

### 寿命の仕組み

- ゲーム内1年（現実約6時間）で1歳加齢
- 初期NPCの年齢: 20〜30歳でランダム設定
- 寿命: 60〜80歳でランダム設定（パラメータ非公開）
- 寿命が近づくと（残り5年以内）:
  - 行動が変化（遺言を残す、後継者を指名する、過去を振り返る）
  - プロンプトに「あなたは老いを感じている。残された時間は多くない」を追加

### 死亡時の処理

- NPCがマップから消える
- 全NPCの記憶に「{名前}が亡くなった」を追加
- 村の歴史に記録
- 死亡NPCの `beliefs` や `proposals` は「遺産」として村の歴史に残る
- 他NPCの会話に影響（「あの人が言ってたことを守ろう」）

### 誕生・次世代

- 関係性スコアが高いNPCペア（+70以上）に子どもが生まれる可能性
- 子どもは両親のパラメータを一部受け継ぐ（遺伝）+ ランダム要素
- 子どもは最初は行動範囲が狭く、成長とともに広がる
- 親の記憶を一部（文化・信仰）として受け継ぐ → 文化の世代間伝承
